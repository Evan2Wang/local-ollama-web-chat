# 本地 Ollama Web Chat 工具产品文档

> 版本：V1.1 基线版
> 项目名：local-ollama-web-chat
> 运行环境：Windows 本地环境
> 底层模型服务：Ollama
> 默认模型：qwen3.6:35b-a3b
> 文档定位：面向 Codex / OpenCode / 本地开发代理的产品需求与开发约束文档

---

## 1. 项目背景

我已经在本地部署了 **Ollama + Qwen3.6 35B**。模型能力本身已经可用，但 Ollama 原生交互体验不足，尤其不适合长期作为个人工作助手使用。

当前主要痛点包括：

1. 历史对话管理不方便；
2. 多轮对话体验不够稳定；
3. Markdown、表格、代码块等富文本展示不够友好；
4. 不支持像 ChatGPT 一样自然地直接粘贴图片、粘贴文件、拖拽文件；
5. 文件上传后缺少清晰的“文件解析 → 对话问答 → 历史沉淀”流程；
6. 本地服务缺少基础访问保护；
7. 常用任务缺少快捷提示词，重复输入成本高；
8. 文件解析质量不可见，模型回答不好时难以判断是模型问题还是文件解析问题；
9. Ollama 连接问题排查不方便，需要内置诊断能力。

因此需要开发一个 **本地轻量化 Web 工具**，底层继续调用本机 Ollama API，重点改善交互体验、会话管理、文件处理、访问保护和诊断能力。

本项目不是重新训练模型，也不是替代 Ollama，而是作为 **Ollama 的本地增强交互层**。

---

## 2. 产品定位

开发一个运行在本地电脑上的轻量级 Web Chat 工具。

一句话定位：

> 一个本地版轻量 ChatGPT 前端，底层调用 Ollama，重点优化历史记录、多轮对话、富文本展示、图片粘贴、文件粘贴、文件问答、快捷提示词和本地访问保护体验。

本项目主要面向个人本地使用，不做复杂多用户系统，不接入云服务，不做在线同步。

---

## 3. 产品版本规划

### 3.1 V1.0 MVP：基础可用

V1.0 目标是先跑通本地聊天主链路：

1. 调用本地 Ollama 模型；
2. 支持多轮对话；
3. 支持历史记录；
4. 支持 Markdown 富文本渲染；
5. 支持直接粘贴图片、拖拽图片、上传图片；
6. 支持直接粘贴文件、拖拽文件、上传文件；
7. 支持文件解析后参与对话；
8. 支持本地 SQLite 保存会话、消息和附件。

### 3.2 V1.1：从可用工具升级为长期可用的本地助手

V1.1 重点完成五个增强功能：

1. Token 字符串认证；
2. 快捷提示词 / Prompt 模板；
3. 附件解析结果预览；
4. 会话搜索；
5. Ollama 诊断页。

V1.1 基线补充约束：

1. 支持数学公式渲染，Markdown 中常见 LaTeX 行内公式和块级公式应正常展示；
2. 支持侧边栏收起/展开，收起后保留新建会话入口；
3. 支持生成中停止输出，前端通过 AbortController 中断当前流式请求；
4. 为保护本地 Ollama，同一时间只允许一个生成任务；切换会话不应额外发起并发 `/api/chat`；
5. 后端对 Ollama chat 调用做进程内串行化，避免多个窗口或误触并发阻塞本地模型；
6. 诊断页 `/api/health/chat` 只做非思考短探针，避免打开诊断页时触发长时间推理；正式聊天仍按 `OLLAMA_THINK` 配置执行；
7. 文件解析默认尽量保留长文档内容，`MAX_FILE_CHARS=1000000`，设为 `0` 时关闭主动截断；
8. UI 风格以深色侧栏、浅色主工作区、轻量卡片和悬浮底部输入区为基线，不做营销式落地页；
9. 快捷提示词行和输入框应保持同宽，输入区周围不放整条实心横栏，聊天内容区域应能完整滚动。

### 3.3 V1.2：专业工作流增强

后续可考虑：

1. 会话导出 Markdown / JSON；
2. 系统提示词 Profile；
3. 图片 OCR；
4. 视觉模型增强；
5. PDF 页码引用；
6. Word / PPT 结构化解析增强；
7. 会议纪要、PPT 大纲、储能资料总结等专用工作流。

### 3.4 V2.0：本地知识库与项目空间

后续可考虑：

1. 本地知识库；
2. 向量检索；
3. 项目空间；
4. 储能资料库；
5. 长期文档沉淀；
6. Agent 自动执行任务。

## 4. 目标用户与典型场景

### 4.1 目标用户

主要用户是我本人，运行在本地 Windows 电脑上，用于日常办公、资料处理、技术学习和汇报材料生成。

### 4.2 典型使用场景

1. 粘贴截图，让模型解释问题；
2. 上传 PDF / Word / PPT / Excel，让模型总结；
3. 把会议纪要、储能资料、技术文档转成结构化笔记；
4. 基于文件内容连续追问；
5. 生成领导汇报材料、PPT 大纲、邮件、需求文档；
6. 保留历史对话，后续可以搜索和复用；
7. 使用快捷提示词快速完成“文件总结”“图片转文字”“压缩成领导汇报版”等任务；
8. 当 Ollama 连接异常时，可以通过诊断页快速定位问题。

## 5. 总体设计原则

1. **本地优先**：所有数据默认保存在本机，不依赖云端服务。
2. **轻量优先**：不引入复杂账号体系、权限体系、云同步或大型 RAG。
3. **可用优先**：先保证聊天、文件上传、历史记录和诊断能力稳定。
4. **可维护优先**：代码结构清晰，方便 Codex / OpenCode 后续增量开发。
5. **可诊断优先**：本地模型服务、后端服务、文件解析状态必须可见。
6. **不编造优先**：基于文件问答时，模型需要明确区分“文件中有的信息”和“无法确认的信息”。
7. **办公场景优先**：重点服务文档总结、截图识别、PPT 大纲、会议纪要、储能资料分析等实际工作流。

## 6. V1.0 功能基线

V1.1 是在 V1.0 基础上增量开发。以下是 V1.0 需要保留和稳定的基础能力。

---

### 6.1 聊天主界面

页面布局参考 ChatGPT：

```text
左侧：会话列表
中间：当前对话区
底部：输入框 + 附件区 + 发送按钮
右上角：模型选择、设置入口
```

聊天区需要支持：

1. 用户消息展示；
2. AI 消息展示；
3. 流式输出；
4. Markdown 渲染；
5. 表格渲染；
6. 代码块高亮；
7. 代码块一键复制；
8. 消息一键复制；
9. 当前消息关联的附件展示；
10. 刷新页面后历史消息仍可恢复。

---

### 6.2 多轮对话

每个会话需要保存完整上下文。

发送消息时，后端应把当前会话最近 N 轮消息组装成 Ollama `/api/chat` 的 `messages` 格式。

建议默认保留最近 20 轮消息，后续可在设置里调整。

消息格式逻辑：

```json
[
  {
    "role": "system",
    "content": "你是一个本地助手，请用中文回答，结构清晰。"
  },
  {
    "role": "user",
    "content": "第一轮问题"
  },
  {
    "role": "assistant",
    "content": "第一轮回答"
  },
  {
    "role": "user",
    "content": "第二轮问题"
  }
]
```

---

### 6.3 历史记录

左侧会话列表需要支持：

1. 新建会话；
2. 自动生成会话标题；
3. 会话重命名；
4. 会话删除；
5. 按更新时间排序；
6. 点击历史会话恢复完整上下文。

会话标题生成方式：

第一版可以直接截取用户第一条消息前 20 个字。后续可增加“让模型自动总结标题”。

---

### 6.4 富文本渲染

AI 输出必须按 Markdown 渲染。

需要支持：

1. 标题；
2. 加粗；
3. 列表；
4. 表格；
5. 代码块；
6. 引用块；
7. 分割线；
8. 链接；
9. 复制按钮。

重点要求：

1. 表格不能错位；
2. 代码块需要可复制；
3. 长文本需要有良好阅读体验；
4. 刷新后 Markdown 渲染效果不丢失。

---

### 6.5 图片输入

图片输入是核心刚需。

必须支持三种方式：

1. `Ctrl + V` 直接粘贴截图；
2. 拖拽图片到聊天窗口；
3. 点击按钮选择图片上传。

支持格式：

```text
png, jpg, jpeg, webp
```

前端行为：

1. 粘贴或拖入图片后，在输入框上方显示缩略图；
2. 支持删除待发送图片；
3. 支持一次发送多张图片；
4. 发送后图片成为当前用户消息的附件；
5. 历史消息中能看到图片缩略图；
6. 点击缩略图可预览大图。

后端处理逻辑：

1. 图片保存到本地 `data/uploads/images/`；
2. 数据库保存图片路径、原始文件名、文件类型、大小；
3. 如果当前模型支持视觉，则将图片转 Base64 后通过 Ollama `images` 数组传入；
4. 如果当前模型不支持视觉，则保留图片附件，并提示用户当前模型无法直接识图；
5. 后续可接入专门的视觉模型用于图片理解。

注意：

> 不要假设所有 Ollama 模型都支持视觉。Ollama API 支持传图，但是否能理解图片取决于具体模型。

---

### 6.6 文件输入

文件输入也是核心刚需。

必须支持三种方式：

1. 直接粘贴文件；
2. 拖拽文件到聊天窗口；
3. 点击按钮选择文件上传。

P0 支持格式：

```text
txt, md, pdf, docx, pptx, xlsx, csv
```

前端行为：

1. 文件进入输入框附件区；
2. 展示文件名、大小、类型；
3. 支持删除待发送文件；
4. 支持一次发送多个文件；
5. 发送后附件显示在用户消息下方；
6. 历史对话中保留附件记录。

后端处理逻辑：

1. 原始文件保存到 `data/uploads/files/`；
2. 解析后的文本保存到 `data/parsed/`；
3. 数据库记录文件状态；
4. 发送给模型时，将解析后的文本拼接到用户问题中；
5. 如果文件太长，只截取前 N 字符，并提示用户内容已截断。

文件状态：

```text
uploaded
parsing
parsed
failed
```

P0 文件解析建议：

```text
txt/md/csv：直接读取文本
pdf：优先使用 pymupdf 或 pypdf
docx：使用 python-docx
pptx：使用 python-pptx
xlsx：使用 openpyxl
```

## 7. 文件参与对话的提示词格式

当用户上传文件并提问时，后端应把内容组织成下面格式：

```text
用户问题：
{user_input}

以下是用户上传的文件内容，请基于这些内容回答：

【文件1：xxx.pdf】
{parsed_text_1}

【文件2：xxx.docx】
{parsed_text_2}

回答要求：
1. 优先基于文件内容回答；
2. 如果文件内容不足，请明确说明；
3. 不要编造文件中没有的信息；
4. 用中文回答；
5. 结构清晰，必要时使用表格。
```

如果文件内容被截断，应追加提示：

```text
注意：该文件内容较长，当前仅截取前 {MAX_FILE_CHARS} 字符参与分析。
```

如果是图片且模型支持视觉：

```json
{
  "role": "user",
  "content": "请解释这张截图的问题",
  "images": ["base64_image_1"]
}
```

## 9. 配置项

`.env` 建议配置：

```env
OLLAMA_BASE_URL=http://127.0.0.1:11434
DEFAULT_MODEL=qwen3.6:35b-a3b
MAX_FILE_CHARS=30000

AUTH_ENABLED=true
APP_TOKEN=change-me-local-token

CHAT_CONTEXT_TURNS=20
STREAM_ENABLED=true
```

配置说明：

### 9.1 OLLAMA_BASE_URL

Ollama 服务地址。

必须默认使用：

```text
http://127.0.0.1:11434
```

不要默认使用：

```text
http://localhost:11434
```

原因：

1. Windows 下 `localhost` 可能解析到 IPv6；
2. 公司网络环境下可能受代理影响；
3. Python `httpx/requests` 可能读取系统代理变量导致本地请求超时。

### 9.2 DEFAULT_MODEL

默认模型名称，必须和 `ollama list` 中显示的模型名完全一致。

示例：

```text
qwen3.6:35b-a3b
```

### 9.3 MAX_FILE_CHARS

文件解析后，最多进入模型上下文的字符数。

例如：

```env
MAX_FILE_CHARS=30000
```

表示每个文件最多取 30000 字符参与模型分析，避免文件过长导致推理变慢、上下文过大或显存压力过高。

### 9.4 AUTH_ENABLED

是否启用 Token 认证。

```env
AUTH_ENABLED=true
```

启用认证。

```env
AUTH_ENABLED=false
```

关闭认证，方便本地调试。

### 9.5 APP_TOKEN

本地访问 Token。

用户首次打开前端时输入该 Token，前端保存到 `localStorage`，后续请求通过 `Authorization` 请求头传给后端。

## 10. 后端 API 设计

---

### 10.1 获取模型列表

```http
GET /api/models
```

后端调用 Ollama：

```http
GET http://127.0.0.1:11434/api/tags
```

返回本地已安装模型。

要求：

1. 不允许拼接成 `/api/api/tags`；
2. 不允许默认使用 `localhost`；
3. 使用 `httpx` 时必须设置 `trust_env=False`；
4. 如果失败，需要返回真实错误信息。

---

### 10.2 创建会话

```http
POST /api/conversations
```

请求：

```json
{
  "model": "qwen3.6:35b-a3b",
  "system_prompt": "你是一个本地中文助手。"
}
```

---

### 10.3 获取会话列表

```http
GET /api/conversations
```

返回：

```json
[
  {
    "id": "xxx",
    "title": "储能资料总结",
    "model": "qwen3.6:35b-a3b",
    "created_at": "...",
    "updated_at": "..."
  }
]
```

---

### 10.4 获取单个会话详情

```http

返回会话信息、消息列表、附件列表。

```

删除会话、消息和附件关联记录。

是否删除本地附件文件，V1.1 可先保留原文件，后续再做清理策略。

### 10.9 发送消息

```http
POST /api/chat
```

请求：

```json
{
  "conversation_id": "xxx",
  "model": "qwen3.6:35b-a3b",
  "content": "帮我总结这个文件",
  "attachment_ids": ["att_xxx"],
  "stream": true
}
```

后端逻辑：

1. 校验 Token；
2. 保存用户消息；
3. 绑定附件到用户消息；
4. 读取附件解析内容；
5. 构建上下文；

8. 完成后保存 AI 消息。

### 11.1 Token 字符串认证

#### 11.1.1 功能定位

Token 认证用于本地服务的基础访问保护。

该功能不是复杂账号系统，不涉及：

1. 用户注册；
2. 多用户管理；
3. 角色权限；
4. 密码找回；
5. 云端登录；
6. 组织权限体系。

它只是一个轻量门禁，类似本地服务访问口令。

#### 11.1.2 配置项

```env
AUTH_ENABLED=true
APP_TOKEN=your-local-secret-token
```

#### 11.1.3 前端交互

1. 首次访问页面时，如果启用认证，显示 Token 登录页；
2. 用户输入 Token；
3. 前端调用 `/api/auth/check` 校验；
4. 校验成功后，将 Token 保存到 `localStorage`；
5. 后续所有 API 请求自动带上：

```http
Authorization: Bearer <token>
```

6. Token 错误时提示“Token 不正确”；
7. 提供“退出登录”按钮，清除本地 Token。

#### 11.1.4 后端逻辑

新增接口：

```http
POST /api/auth/check
```

请求：

```json
{
  "token": "xxx"
}
```

返回：

```json
{
  "ok": true
}
```

认证规则：

1. `AUTH_ENABLED=false` 时跳过认证；
2. `AUTH_ENABLED=true` 时，除白名单接口外，其余接口必须校验 Token；
3. Token 从请求头读取：

```http
Authorization: Bearer <token>
```

4. Token 错误返回：

```http
401 Unauthorized
```

#### 11.1.5 白名单接口

以下接口可不要求 Token：

```text
POST /api/auth/check
GET /api/health/config
GET /api/health/ollama
```

是否开放 `/api/health/chat` 可配置。若担心被滥用，可要求认证。

### 11.2 快捷提示词 / Prompt 模板

#### 11.2.1 功能定位

快捷提示词用于减少重复输入，提高高频任务效率。

典型场景：

1. 文件总结；
2. 图片转文字；
3. 提取表格；
4. 生成 PPT 大纲；
5. 压缩成领导汇报版；
6. 翻译成中文；
7. 提取待办事项；
8. 检查逻辑问题。

#### 11.2.2 前端交互

在输入框上方或侧边增加快捷提示词区域。

交互要求：

1. 展示常用提示词按钮；
2. 点击按钮后，将模板内容插入输入框；
3. 不要自动发送；
4. 用户可以二次修改后再发送；
5. 模板内容较长时，可以弹出预览或直接填入输入框；
6. 支持模板新增、编辑、删除、启用/禁用。

#### 11.2.3 内置模板

##### 文件总结

```text
请总结当前上传的文件，输出：
1. 核心结论
2. 关键事实
3. 重要数据
4. 风险与问题
5. 后续建议

要求：
- 优先基于文件内容回答
- 不要编造文件中没有的信息
- 结构清晰，必要时使用表格
```

##### 图片转文字

```text
请识别图片中的文字，并尽量保持原始排版结构。
如果图片中有表格，请尽量转换为 Markdown 表格。
如果有无法识别的内容，请明确标注“无法识别”。
```

##### 提取表格

```text
请从当前文件或图片中提取表格信息，并转换为 Markdown 表格。
如果原文没有明确表格，请根据内容整理成结构化表格。
```

##### 生成 PPT 大纲

```text
请将当前内容整理成适合领导汇报的 PPT 大纲，要求：
1. 结论先行
2. 每页一个明确标题
3. 每页 3-5 个要点
4. 风格务实，不要空话
5. 尽量使用“结论 + 支撑事实 + 下一步动作”的结构
```

##### 压缩成领导汇报版

```text
请将当前内容压缩成领导汇报版，要求：
1. 先给一句话结论
2. 保留关键事实和数据
3. 删除过程性、重复性、口语化表达
4. 语气正式、简洁、务实
5. 如有风险或待决策事项，请单独列出
```

##### 翻译成中文

```text
请将当前内容翻译成中文，要求：
1. 准确表达原意
2. 保留专业术语
3. 语气自然，适合商务或技术场景
4. 如有歧义，请在译文后说明
```

##### 提取待办事项

```text
请从当前内容中提取待办事项，输出表格：
| 序号 | 待办事项 | 责任方 | 截止时间 | 优先级 | 备注 |

如果内容中没有明确责任方或时间，请填“待确认”。
```

##### 检查逻辑问题

```text
请检查当前内容的逻辑问题，重点关注：
1. 结论是否清晰
2. 论据是否支撑结论
3. 是否有表达重复
4. 是否有事实不完整或需要确认的地方
5. 是否适合给领导或客户阅读

请输出：
- 主要问题
- 修改建议
- 优化后的版本
```

### 11.3 附件解析结果预览

#### 11.3.1 功能定位

附件解析预览用于提升文件问答的可信度和可调试性。

目标是让用户知道：

1. 文件是否上传成功；
2. 文件是否解析成功；
3. 解析出来的文本是什么；
4. 是否被 `MAX_FILE_CHARS` 截断；
5. 实际送入模型的内容有多少；
6. 如果模型回答不好，是模型问题还是解析问题。

#### 11.3.2 前端交互

用户点击消息中的附件卡片后，打开文件详情面板。

文件详情面板显示：

1. 文件名；
2. 文件类型；
3. 文件大小；
4. 上传时间；
5. 解析状态；
6. 解析后文本预览；
7. 是否被截断；
8. 原始解析字符数；
9. 实际进入模型的字符数；
10. 错误信息；
11. 重新解析按钮。

#### 11.3.3 后端逻辑

文件解析后记录：

```text
status
error_message
parsed_text_path
is_truncated
original_chars
used_chars
```

如果解析失败，必须记录可读错误信息，而不是只显示 `failed`。

#### 11.3.4 API

```http
GET /api/attachments/{attachment_id}
POST /api/attachments/{attachment_id}/reparse
```

### 11.5 Ollama 诊断页

#### 11.5.1 功能定位

Ollama 诊断页用于快速判断本地模型服务是否正常。

需要解决的问题：

1. Ollama 是否在线；
2. Ollama 地址是否配置正确；
3. `/api/tags` 是否能访问；
4. `/api/chat` 是否能调用；
5. 默认模型是否存在；
6. 当前配置是否和 `.env` 一致；
7. 是否还有 `localhost`、代理、timeout 等问题。

#### 11.5.2 诊断页展示内容

诊断页显示：

1. 后端状态；
2. Ollama Base URL；
3. 默认模型；
4. MAX_FILE_CHARS；
5. AUTH_ENABLED；
6. Ollama 在线状态；
7. 本地模型列表；
8. `/api/tags` 测试结果；
9. `/api/chat` 测试结果；
10. 最近一次错误信息。

#### 11.5.3 技术要求

后端访问 Ollama 时必须遵守：

1. 默认使用：

```text
http://127.0.0.1:11434
```

2. 不要默认使用：

```text
http://localhost:11434
```

3. 不允许拼接成：

```text
http://127.0.0.1:11434/api/api/tags
```

4. 如果使用 `httpx`，必须设置：

```python
trust_env=False
```

5. 如果使用 `requests`，必须避免系统代理影响本地请求。

#### 11.5.4 API

```http
GET /api/health/config
GET /api/health/ollama
GET /api/health/chat
```

##### GET /api/health/config

返回：

```json
{
  "ok": true,

  "default_model": "qwen3.6:35b-a3b",
  "max_file_chars": 30000,
  "auth_enabled": true
}
```

##### GET /api/health/ollama

返回：

```json
{
  "ok": true,
  "url": "http://127.0.0.1:11434/api/tags",
  "status_code": 200,
  "models": [
    "qwen3.6:35b-a3b",
    "gemma4:31b"
  ]
}
```

##### GET /api/health/chat

返回：

```json
{
  "ok": true,
  "url": "http://127.0.0.1:11434/api/chat",
  "model": "qwen3.6:35b-a3b",
  "response_preview": "你好！"
}
```

失败时返回：

```json
{
  "ok": false,
  "error": "ReadTimeout",
  "url": "http://127.0.0.1:11434/api/tags",
  "status_code": null,
  "detail": "..."
}
```

## 17. 关键风险与约束

### 17.1 Token 认证风险

Token 存在浏览器 `localStorage`，只适合作为本地轻量门禁，不适合作为企业级安全方案。

### 17.2 文件解析风险

PDF / PPT / Excel 的解析质量可能不稳定，因此必须提供解析预览，方便用户判断解析结果是否可信。

### 17.3 大文件风险

文件过长会导致上下文过大、推理变慢、显存压力上升，因此必须通过 `MAX_FILE_CHARS` 控制进入模型的内容长度。

### 17.4 Ollama 连接风险

Windows 本地环境中，`localhost`、IPv6、代理环境变量、timeout 都可能导致后端连接 Ollama 失败。因此后端必须默认使用 `127.0.0.1`，并关闭环境代理影响。

### 17.5 模型能力风险

不是所有 Ollama 模型都支持视觉。图片输入和图片理解是两个不同能力：

1. 图片上传和保存：前端/后端能力；
2. 图片理解：模型能力。

如果当前模型不支持视觉，应明确提示，而不是假装已经理解图片。
