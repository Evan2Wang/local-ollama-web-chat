# 本地 Ollama Web Chat 工具 MVP 需求文档

## 1. 项目背景

我已经在本地部署了 **Ollama + Qwen3.6 35B**，模型能力本身可用，但 Ollama 原生交互体验不足，主要问题包括：

1. 不方便管理历史对话；
2. 多轮对话体验弱；
3. Markdown / 表格 / 代码块等富文本展示不够友好；
4. 不支持像 ChatGPT 一样自然地直接粘贴图片、粘贴文件、拖拽文件；
5. 文件上传后无法形成顺畅的“文件解析 → 对话问答 → 历史沉淀”流程。

因此需要开发一个 **本地轻量化 Web 工具**，底层仍然调用本机 Ollama API，只改善交互体验、会话管理和文件处理能力。

Ollama 本地 API 默认可通过 `http://127.0.0.1:11434/api` 访问，并支持 `/api/chat` 聊天接口、流式输出，以及 `/api/tags` 获取本地模型列表。对于支持视觉的模型，Ollama REST API 可通过 `images` 数组传入 Base64 图片。([docs.ollama.com][1])

---

## 2. 产品定位

开发一个运行在本地电脑上的轻量级 Web Chat 工具。

一句话定位：

> 一个本地版轻量 ChatGPT 前端，底层调用 Ollama，重点优化历史记录、多轮对话、富文本展示、图片粘贴、文件粘贴和文件问答体验。

本项目不是重新训练模型，也不是替代 Ollama，而是作为 **Ollama 的本地增强交互层**。

---

## 3. MVP 核心目标

第一版不要做复杂知识库，不要做大型 RAG，不要做账号系统。

MVP 只聚焦六件事：

1. 支持调用本地 Ollama 模型；
2. 支持多轮对话；
3. 支持历史记录；
4. 支持 Markdown 富文本渲染；
5. 支持直接粘贴图片、拖拽图片、上传图片；
6. 支持直接粘贴文件、拖拽文件、上传文件，并能解析为文本后参与对话。

---

## 4. 目标用户

主要用户是我本人。

典型使用场景：

1. 粘贴截图，让模型解释问题；
2. 上传 PDF / Word / PPT / Excel，让模型总结；
3. 把会议纪要、储能资料、技术文档转成结构化笔记；
4. 基于文件内容连续追问；
5. 生成领导汇报材料、PPT大纲、邮件、需求文档；
6. 保留历史对话，后续可以搜索和复用。

---

## 5. P0 功能范围

### 5.1 聊天主界面

页面布局参考 ChatGPT：

左侧：会话列表
中间：当前对话区
底部：输入框 + 附件区 + 发送按钮
右上角：模型选择、设置入口

聊天区需要支持：

1. 用户消息展示；
2. AI 消息展示；
3. 流式输出；
4. Markdown 渲染；
5. 表格渲染；
6. 代码块高亮；
7. 代码块一键复制；
8. 消息一键复制；
9. 当前消息关联的附件展示。

---

### 5.2 多轮对话

每个会话需要保存完整上下文。

发送消息时，后端应把当前会话最近 N 轮消息组装成 Ollama `/api/chat` 的 `messages` 格式。

建议默认保留最近 20 轮消息，后续在设置里可调整。

消息格式逻辑：

```json
[
  {
    "role": "system",
    "content": "你是一个本地助手，请用中文回答，结构清晰。"
  },
  {
    "role": "user",
    "content": "第一轮问题"
  },
  {
    "role": "assistant",
    "content": "第一轮回答"
  },
  {
    "role": "user",
    "content": "第二轮问题"
  }
]
```

---

### 5.3 历史记录

左侧会话列表需要支持：

1. 新建会话；
2. 自动生成会话标题；
3. 会话重命名；
4. 会话删除；
5. 按更新时间排序；
6. 点击历史会话恢复完整上下文。

会话标题生成方式：

第一版可以直接截取用户第一条消息前 20 个字。

后续再增加“让模型自动总结标题”。

---

### 5.4 富文本渲染

AI 输出必须按 Markdown 渲染。

需要支持：

1. 标题；
2. 加粗；
3. 列表；
4. 表格；
5. 代码块；
6. 引用块；
7. 分割线；
8. 链接；
9. 复制按钮。

重点是表格体验要好，因为我经常让模型输出对比表、需求表、汇报材料结构表。

---

### 5.5 图片输入

这是核心刚需。

必须支持三种方式：

1. `Ctrl + V` 直接粘贴截图；
2. 拖拽图片到聊天窗口；
3. 点击按钮选择图片上传。

支持格式：

```text
png, jpg, jpeg, webp
```

前端行为：

1. 粘贴或拖入图片后，在输入框上方显示缩略图；
2. 支持删除待发送图片；
3. 支持一次发送多张图片；
4. 发送后图片成为当前用户消息的附件；
5. 历史消息中能看到图片缩略图；
6. 点击缩略图可预览大图。

后端处理逻辑：

1. 图片保存到本地 `data/uploads/images/`；
2. 数据库保存图片路径、原始文件名、文件类型、大小；
3. 如果当前模型支持视觉，则将图片转 Base64 后通过 Ollama `images` 数组传入；
4. 如果当前模型不支持视觉，则先保留图片附件，并提示用户当前模型无法直接识图；
5. 后续可以接入专门的视觉模型，例如本地 VL 模型，用于图片理解。

注意：不要假设所有 Ollama 模型都支持视觉。Ollama API 支持传图，但是否能理解图片取决于具体模型。([docs.ollama.com][2])

---

### 5.6 文件输入

这也是核心刚需。

必须支持三种方式：

1. 直接粘贴文件；
2. 拖拽文件到聊天窗口；
3. 点击按钮选择文件上传。

P0 支持格式：

```text
txt, md, pdf, docx, pptx, xlsx, csv
```

前端行为：

1. 文件进入输入框附件区；
2. 展示文件名、大小、类型；
3. 支持删除待发送文件；
4. 支持一次发送多个文件；
5. 发送后附件显示在用户消息下方；
6. 历史对话中保留附件记录。

后端处理逻辑：

1. 原始文件保存到 `data/uploads/files/`；
2. 解析后的文本保存到 `data/parsed/`；
3. 数据库记录文件状态：

   * uploaded
   * parsing
   * parsed
   * failed
4. 发送给模型时，将解析后的文本拼接到用户问题中；
5. 如果文件太长，只截取前 N 字符，并提示用户内容已截断。

P0 文件解析建议：

```text
txt/md/csv：直接读取文本
pdf：优先使用 pymupdf 或 pypdf
docx：使用 python-docx
pptx：使用 python-pptx
xlsx：使用 openpyxl
```

## 6. 文件参与对话的提示词格式

当用户上传文件并提问时，后端应把内容组织成下面格式：

```text
用户问题：
{user_input}

以下是用户上传的文件内容，请基于这些内容回答：

【文件1：xxx.pdf】
{parsed_text_1}

【文件2：xxx.docx】
{parsed_text_2}

回答要求：
1. 优先基于文件内容回答；
2. 如果文件内容不足，请明确说明；
3. 不要编造文件中没有的信息；
4. 用中文回答；
5. 结构清晰，必要时使用表格。
```

如果是图片且模型支持视觉：

```json
{
  "role": "user",
  "content": "请解释这张截图的问题",
  "images": ["base64_image_1"]
}
```

## 7. 推荐技术架构

### 7.1 前端

建议使用：

```text
React + Vite + TypeScript
```

前端主要模块：

```text
src/
  components/
    ChatWindow.tsx
    MessageBubble.tsx
    Composer.tsx
    AttachmentPreview.tsx
    Sidebar.tsx
    MarkdownRenderer.tsx
    ModelSelector.tsx
  pages/
    Home.tsx
  api/
    client.ts
  types/
    chat.ts
```

关键前端能力：

1. 监听 paste 事件，识别图片和文件；
2. 监听 drop 事件，支持拖拽上传；
3. 用 FormData 上传附件；
4. 用 EventSource 或 fetch stream 接收流式回答；
5. 用 Markdown 渲染 AI 消息。

---

### 7.2 后端

建议使用：

```text
FastAPI + SQLite + SQLAlchemy
```

后端模块：

```text
backend/
  app/
    main.py
    config.py
    database.py
    models.py
    schemas.py
    routers/
      chat.py
      conversations.py
      attachments.py
      models.py
    services/
      ollama_client.py
      file_parser.py
      title_service.py
      context_builder.py
    storage/
      file_store.py
```

---

### 7.3 数据库

使用 SQLite 即可，文件名：

```text
data/app.db
```

核心表设计：

#### conversations

```sql
id TEXT PRIMARY KEY
title TEXT
created_at DATETIME
updated_at DATETIME
model TEXT
system_prompt TEXT
```

#### messages

```sql
id TEXT PRIMARY KEY
conversation_id TEXT
role TEXT
content TEXT
created_at DATETIME
token_count INTEGER
```

#### attachments

```sql
id TEXT PRIMARY KEY
message_id TEXT
conversation_id TEXT
filename TEXT
file_type TEXT
mime_type TEXT
size INTEGER
storage_path TEXT
parsed_text_path TEXT
status TEXT
created_at DATETIME
```

## 8. 后端 API 设计

### 8.1 获取模型列表

```http
GET /api/models
```

后端调用 Ollama：

```http
GET http://127.0.0.1:11434/api/tags
```

返回本地已安装模型。

---

### 8.2 创建会话

```http
POST /api/conversations
```

请求：

```json
{
  "model": "qwen3.6:35b",
  "system_prompt": "你是一个本地中文助手。"
}
```

---

### 8.3 获取会话列表

```http
GET /api/conversations
```

返回：

```json
[
  {
    "id": "xxx",
    "title": "储能资料总结",
    "model": "qwen3.6:35b",
    "created_at": "...",
    "updated_at": "..."
  }
]
```

---

### 8.4 获取单个会话详情

```http
GET /api/conversations/{conversation_id}
```

返回会话信息、消息列表、附件列表。

---

### 8.5 上传附件

```http
POST /api/attachments
```

使用 `multipart/form-data`。

字段：

```text
conversation_id
files[]
```

返回：

```json
[
  {
    "id": "att_xxx",
    "filename": "xxx.pdf",
    "status": "parsed"
  }
]
```

---

### 8.6 发送消息

```http
POST /api/chat
```

请求：

```json
{
  "conversation_id": "xxx",
  "model": "qwen3.6:35b",
  "content": "帮我总结这个文件",
  "attachment_ids": ["att_xxx"],
  "stream": true
}
```

返回流式输出。

后端逻辑：

1. 保存用户消息；
2. 绑定附件到用户消息；
3. 解析附件内容；
4. 构建上下文；
5. 调用 Ollama `/api/chat`；
6. 流式返回 AI 回复；
7. 完成后保存 AI 消息。

## 10. MVP 验收标准

### 10.1 基础聊天

必须通过：

1. 打开 Web 页面；
2. 自动读取本地 Ollama 模型列表；
3. 选择 `qwen3.6:35b`；
4. 输入问题；
5. 模型正常流式回答；
6. 刷新页面后历史记录仍存在。

---

### 10.2 多轮对话

必须通过：

第一轮：

```text
请帮我总结储能 Trading 模式。
```

第二轮：

```text
压缩成领导汇报版。
```

第三轮：

```text
改成三页 PPT 结构。
```

模型必须能理解上下文，而不是把每一轮当成独立问题。

---

### 10.3 图片粘贴

必须通过：

1. 截图；
2. 在输入框 `Ctrl + V`；
3. 页面显示图片缩略图；
4. 点击发送；
5. 历史消息中显示图片；
6. 如果模型支持视觉，则模型能基于图片回答；
7. 如果模型不支持视觉，则明确提示当前模型无法直接识图。

---

### 10.4 文件粘贴 / 拖拽

必须通过：

1. 拖入一个 PDF；
2. 页面显示文件卡片；
3. 输入“帮我总结这个文件”；
4. 后端解析 PDF；
5. 模型基于文件内容回答；
6. 对话刷新后，文件附件和回答仍然保留。

---

### 10.5 Markdown 渲染

必须通过：

让模型输出：

```text
请用表格对比 Tolling 和 Merchant 模式。
```

页面需要正确展示 Markdown 表格，而不是纯文本乱排。

---

## 11. 第一版不要做的内容

MVP 暂时不要做：

1. 用户登录；
2. 云同步；
3. 多用户权限；
4. 复杂知识库；
5. 向量数据库；
6. 文件夹批量知识库导入；
7. Agent 自动执行；
8. PPT 自动生成；
9. Word 导出；
10. 插件市场。

这些放到后续版本。
